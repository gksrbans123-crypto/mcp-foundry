# G-A: PlayMCP OAuth 요구 조사 결론

> 계획서 §0.2 선결 과제 G-A: "취급 데이터(잡·서버 메타데이터·owner 매핑)가 가이드상 '개인정보'로 표준 OAuth를 강제하는지 확정" — 이 문서가 그 결론이다.
> 참고: 계획서 §9(AuthN 결정 트리), `.omc/plans/mcp-creator-plan.md`

## 결론 (TL;DR)

**OAuth는 강제되지 않는다. 커스텀 헤더(Key/Token) 방식이 PlayMCP가 공식적으로 동등하게 허용하는 대안이다.**
→ 계획서 §9 결정트리의 **NO 분기** 채택: **impl A = SignedOwnerToken(커스텀 헤더)** 를 v1 AuthN 구현으로 확정한다. AuthN은 인터페이스(`verify(req) -> userId`) 뒤에 은닉해 OAuth(impl B)로 무중단 전환 가능하게 유지한다(계획서 원안 그대로).

---

## 1. 결정 트리 (조사 결과 반영)

```
Q1. PlayMCP 개발 가이드가 "사용자 인증이 필요한 경우" OAuth를 유일한 방식으로 강제하는가?
 └─ NO. 가이드 원문: "OAuth 인증 혹은 커스텀 헤더 방식을 지원해야 합니다." [출처 1]
    → 두 방식은 문서상 대등한 대안. 어느 쪽을 선택할지는 서비스 구현자 재량.

Q2. 우리가 취급하는 데이터(잡/서버 메타데이터, owner 매핑=opaque token hash)가
    심사 정책의 "개인정보" 금지/제한 항목에 해당해 OAuth를 사실상 강제받는가?
 └─ NO. 심사 정책이 명시적으로 금지하는 항목은 다음 6종뿐이다 [출처 2]:
        주민등록번호 · 운전면허번호 · 여권번호 · 외국인등록번호 · 카드번호 · 계좌번호
    우리 데이터 모델(User.auth_ref, Job.input/parsed_spec, Server 메타데이터)은
    이 6종 어디에도 해당하지 않는다. "Tool의 기능과 무관한 개인정보" 수집 금지 조항도
    우리가 수집하는 데이터가 전부 서비스 기능(잡 생성/조회/서버 관리)에 직접 필요한
    범위 내이므로 저촉되지 않는다.

Q3. 심사 정책이 "인증 정보"를 다루는 방식 자체가 OAuth를 우대하는가?
 └─ NO. 정책 원문이 두 방식을 병렬 취급: "OAuth 인증이나 Token/Key 입력 등을 통해
    획득한 인증 정보는 Tool의 사용 목적 외에 다른 용도로 사용해서는 안 됩니다." [출처 2]
    → 의무는 "목적 외 사용 금지"이지 "OAuth를 쓰라"가 아님.

⇒ 종합: G-A 조건(개인정보 취급 → 표준 OAuth 강제)은 성립하지 않는다.
⇒ 채택: impl A (SignedOwnerToken, 커스텀 헤더). 계획서 §9 원안 그대로 진행.
```

## 3. 별도 유의사항 (OAuth와 무관하지만 구현 시 반영 필요)

1. **401 응답 의무** (심사 정책 "HTTP 응답" 항): 인증 정보 부재/만료 시 커스텀 헤더 방식이라도 401 반환 필요 → `apps/creator-mcp` AuthN 미들웨어 요구사항에 반영할 것.
2. **"kakao" 명칭 금지**: MCP Server Name/Tool Name에 "kakao"를 prefix/suffix로 사용 불가 — 계획서 SERVICE_NAME 정책과 이미 정합. (본 조사에서 재확인됨, [출처 2])
3. **Tool 개수 권장 3~20개** (심사 정책 원문은 "3~20"로, 계획서 §8의 "3~10 권장, >20 하드 차단"보다 상한이 완화되어 있음) — **불일치 발견**: 계획서는 20개 초과를 하드 차단 기준으로 잡았는데, 이는 심사 정책의 "3~20 권장" 상한과 일치한다(20 초과 = 권장 범위 밖). 정합성 있음, 추가 조치 불필요.
4. **응답 크기 24k 초과 시 에러 처리** — 계획서에 반영 필요 여부, `packages/validator` 담당자(worker-4) 확인 권고.
5. **동일/유사 MCP 반복 등록 제한** — refine/redeploy 흐름에서 slug 변경만으로 재등록하는 패턴이 "명칭·문구·출력형식만 변경"으로 오인되지 않도록 유의.

## 4. 확인 불가 항목 — 사용자(공모전 참가자) 직접 확인 체크리스트

아래는 공개 웹 문서(Notion 이용가이드/심사정책, 카카오 기술 블로그)로 교차 확인했으나, **자동화된 조사만으로는 100% 확정할 수 없는 항목**이다. 최종 제출 전 PlayMCP 개발자 콘솔/디스코드에서 직접 확인 권고:

- [ ] **관리자 재량 조항 리스크**: 심사 정책 전반에 "관리자 판단에 따라"라는 재량 조항이 반복된다. 커스텀 헤더 인증 채택이 이 재량 심사에서 이번 공모전(MCP Player 10)에 한해 추가 요구사항 없이 통과되는지는 문서만으로 100% 보장할 수 없다. → **PlayMCP 디스코드(개발자 문의 채널)에 "잡/서버 메타데이터 + owner 매핑을 커스텀 헤더 토큰으로 식별하는 구조가 심사 기준상 문제없는지"를 사전 질의할 것을 권고.**
- [ ] **"개인정보 제3자 제공 동의" 화면 요구 여부**: 개발 가이드 원문에 "개인정보를 카카오로 전달하는 것에 대해 사용자에게 '개인정보 제3자 제공 동의'를 받는 화면 구성을 권장"한다는 내용이 확인됨. 이는 **OAuth 경로에서 우리가 Authorization Server 역할을 하며 카카오(PlayMCP)로 PII를 전달하는 시나리오**에 해당하는 것으로 읽히며, 커스텀 헤더 경로(impl A)에서는 해당 없음이 유력하나, 원문 전체(가이드 문서 전문)를 자동화 도구(jina.ai 리더)를 통해 요약본으로만 확인했으므로 **로그인한 브라우저로 원문 Notion 문서를 직접 열람해 문맥을 한 번 더 확인할 것을 권고**한다.
- [ ] **"자사 서비스가 아닌 제3의 서비스 인증 정보" 조항의 적용 범위**: 우리는 자체 발급 SignedOwnerToken만 사용하므로 해당 없음으로 판단되나, "제3의 서비스"의 정의(예: 우리가 향후 OpenAPI 업스트림의 API 키를 대리 보관하는 경우 포함 여부)는 명문화되어 있지 않다. Generator가 사용자 제공 API 키/OpenAPI 인증정보를 다루게 될 경우(§4, 생성된 서버 쪽) 재확인 필요.
- [ ] **MCP Player 10 공모전 전용 추가 규정 존재 여부**: 본 조사에서는 PlayMCP 플랫폼의 일반 이용가이드/심사정책만 확인했다. 공모전 페이지(`https://b.kakao.com/views/PlayMCP/MCP_Player_10`)에 안내된 것 외의 별도 기술 규정집이 있는지는 확인하지 못했다 — 공모전 사무국 안내를 재확인할 것.

## 5. 최종 권고

- **v1 구현**: `packages/shared`의 AuthN 인터페이스(`verify(req) -> userId`) 뒤에 **impl A(SignedOwnerToken, 커스텀 헤더)** 구현. 최초 호출 시 토큰 발급, 대시보드 URL 내장, 응답은 마크다운.
- **필수 반영**: 인증 실패/만료 시 401 반환(§3-1), kakao 명칭 금지 재확인(§3-2, 기존 정책과 정합).
- **인터페이스 유지**: impl B(OAuth, Redirect URI 스텁 `https://playmcp.kakao.com/api/v1/applied-mcps/{mcpId}/authorize/oauth:callback` 문서화만 해두고 미구현)로 **무중단 전환 가능성**은 계획서 원안대로 남겨둔다 — §4 체크리스트 중 하나라도 부정적으로 확인되면 즉시 impl B로 전환.
- 본 문서는 **G-A 게이트를 해제(unblock)** 하는 근거 문서로 사용한다. AuthN 착수(Task #5, `apps/creator-mcp`)는 이 결론에 따라 impl A로 바로 진행 가능.
